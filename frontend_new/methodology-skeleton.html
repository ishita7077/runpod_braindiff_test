Methodology · Skeleton

How we detect content changes.

Every video has structural beats — moments where the topic shifts, a scene cuts, the audio swells or drops. The Structural Skeleton surfaces these as three lanes and asks one question: do they coincide? When text + visual + audio change together, you have tight pacing. When they're misaligned, your content's rhythm is working against itself.

v1 scope. This page documents what's shipping today and what's deferred. The current detector uses only data the worker already produces (transcript segments, audio waveform, video keyframes). The richer detection paths described in the original product spec — foundation-model intermediate embeddings, a 4-class audio classifier, per-shift LLM summaries — are deferred to v2 and explicitly listed below. The bar for shipping is "real, not faked": every event you see came from real signal in your upload.

The three lanes

Text — topic shifts
Visual — scene cuts
Audio — energy + silence

Text — TF-IDF topic shifts

The worker emits WhisperX-aligned phrase segments (meta.transcript_segments_a/b). For each consecutive pair of phrases we compute a TF-IDF vector and take the cosine distance. A topic shift = distance ≥ 0.55 (English text; conservative because phrases are short). Each event carries the first sentence of the phrase before and after — what the user actually heard, not an invented summary.

for i in 1..len(transcript_segments) - 1: v_prev = tfidf(tokens(segment[i-1])) v_next = tfidf(tokens(segment[i])) d = 1 - cosine_similarity(v_prev, v_next) if d >= 0.55: emit topic_shift @ segment[i].start before_summary = first sentence of segment[i-1] after_summary = first sentence of segment[i]

Audio — energy spikes + long silences

The worker emits a 200-bin RMS amplitude envelope (meta.media_features.waveform_a/b). We detect two kinds of events: energy spikes (a bin's RMS is ≥ 2.5 standard deviations above the local rolling mean, with non-maximum suppression so a single loud moment doesn't fire multiple adjacent events) and long silences (runs of bins with RMS < 0.05 lasting ≥ 1 second).

for each bin i in waveform: m, s = rolling_mean_std(waveform, around=i, window=8) z = (waveform[i] - m) / s if z >= 2.5 and waveform[i] >= 0.10: emit energy_spike @ i (with NMS over ±0.5s) walk runs of consecutive bins with RMS < 0.05: if run_duration >= 1.0s: emit silence_start @ run_start, with duration

Visual — scene-detected keyframes

For video the worker already extracts keyframes at scene boundaries via ffmpeg scdet, with uniform fallback sampling when scdet returns nothing. We surface each keyframe time as a visual event. Magnitude scales with the cadence (1 / gap to the previous keyframe) — denser keyframes mean faster cutting.

for each keyframe in meta.media_features.keyframes_a/b: emit scene_cut @ keyframe.time magnitude = 1 / max(1, gap_to_previous_keyframe)

Cross-modal alignment

Two events from different lanes within 1 second of each other count as aligned. We collect every event across the three lanes, mark coincidences, and compute:

alignment_score = aligned_events / total_events

Aligned moments render as vertical bars in the fourth lane. Unmatched events become "misaligned moments" with a leading_modality field — the lane that fired without a companion. This is the simplest possible metric for whether your content's rhythm is internally synchronised.

Honest scope

  • The text detector runs on WhisperX phrases. WhisperX is good but not perfect — for music or non-speech audio it may return nothing, in which case the text lane will be empty.
  • The audio detector is a signal-processing heuristic, not a classifier. It can't tell "spike caused by a door slamming" from "spike caused by the speaker getting excited." Both register as energy spikes.
  • The visual detector relies on ffmpeg scene detection. Slow cross-fades may miss; hard cuts always register.
  • The 1-second alignment window is deliberately loose. Tighter windows (0.3 s) would suppress real coincidences caused by per-second TRIBE prediction granularity.

What's deferred to v2

The original spec proposed a richer detection pipeline using intermediate outputs from the foundation models inside TRIBE v2 (LLaMA 3.2-3B for text, V-JEPA2 for video, Wav2Vec-BERT for audio). We deliberately did not ship those in v1.

Foundation-model intermediate embeddings (LLaMA / V-JEPA2 / Wav2Vec-BERT)

TRIBE caches these on disk via the upstream neuralset.extractors library during prediction, but surfacing them through the worker requires plumbing into a separate package's cache conventions. Reach: medium-rich event detection from the actual encoder hidden states (deeper than TF-IDF for text, smoother than scene detection for video). Cost: ~1–2 days of careful library work.

4-class audio classifier (speech / music / silence / noise)

The spec asked for a classifier head trained on AudioSet or a fine-tuned ~30 minutes of labelled data. We don't have that labelled data, and using a pretrained model without scope validation would risk silently mislabeling content. The v1 energy-based detector covers the common case (loud → quiet, speech → silence) without requiring a corpus we haven't validated.

Per-topic-shift LLM summaries (6-word "before" / "after")

The original spec wanted each topic shift annotated with a 6-word LLM-generated summary of the segments on either side. That adds an extra LLM inference per topic shift, additional prompt-engineering effort, and modest value over showing the first sentence of each segment — which is what you actually heard. v1 ships first-sentence pairs; v2 can layer summaries on top if usage data shows the longer text is the bottleneck.

All three deferments are tracked in backend/structural_skeleton.py docstring and will ship as separate features when their respective pre-requisites (corpus, embedding plumbing, LLM-call budget) land.
